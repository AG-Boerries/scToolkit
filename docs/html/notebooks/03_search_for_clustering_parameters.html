Single-Cell Clustering & Dimensionality Reduction Pipeline

SUMMARY

  1. Initialize environment and load the preprocessed AnnData object

  2. Perform Principal Component Analysis (PCA) and determine optimal components

  3. Calculate cell-type activity scores (ORA) for UMAP evaluation

  4. Execute grid search for UMAP hyperparameters (n_neighbors, min_dist)

  5. Optimize Leiden clustering resolution for broad cell type definition

  6. Finalize neighborhood graph and embeddings based on selected parameters

  7. Validate clustering via Jaccard similarity, QC metrics, and marker gene plots

  8. Save the final clustered object

1. Environment & Data Loading

  • Load scToolkit and other libraries

  • Load 02_tonsil_blood_DISCO_clustering_preprocessed.h5ad

  • Note on Determinism: Acknowledge that GPU-accelerated methods (UMAP) are non-deterministic; parameters must be saved, or embeddings explicitly stored to ensure reproducibility.

2. PCA & Variance Analysis

  • Data Prep: Set adata.X to the scaled layer (Z-scores) for PCA.

  • Execution: Run PCA via sc_code.cluster_SC_scanpy_like.

  • Component Selection: Use sc_code.collect_n_pcs_for_variance_sums to find the number of PCs explaining 85–99% variance.

3. Pre-computation for Evaluation

  • Objective: Generate labels to evaluate UMAP separation during hyperparameter tuning.

  • Method: Use sc_utils.add_cell_type_idxmax_from_significant with method="ora" (Over-Representation Analysis).

  • Result: Create cell_type_estimation_idxmax_ora in obs to color-code cells by their predicted identity (e.g., T cells, B cells) based on marker enrichment.

4. UMAP Hyperparameter Tuning

  • Grid Search: Run sc_code.search_for_neighbor_params to test combinations of:

    • n_pcs: [4, 14, 24]

    • n_neighbors: [15, 25, 50, 75]

    • min_dist: [0.2, 0.3, 0.5]

  • Evaluation Criteria:

    • Batch Effect: Do samples from different studies overlap?

    • Biological Structure: Do known cell types (from ORA scores) cluster logically without artificial fragmentation?

  • Selection: Optimal parameters identified as ``n_pcs=4``, ``n_neighbors=50``, ``min_dist=0.3`` for our use case.

5. Leiden Resolution Search

  • Grid Search: Use sc_code.search_for_leiden_resolution to test resolutions between 0.01 and 0.02.

  • Objective: Find a resolution that yields a stable number of clusters corresponding to major lineages (immune vs. granulocytes).

  • Selection: Resolution 0.012 selected for broad clustering.

6. Final Graph Construction

  • Parameter Application: Update adata.uns["config"] with the chosen values.

  • Re-run: Execute sc_code.cluster_SC_scanpy_like with run_neighbors=True to build the deterministic neighborhood graph using the optimized parameters.

  • Embedding Storage: Overwrite adata.obsm["X_umap"] and adata.obs["leiden"] with the best results from the search dictionaries to ensure the visual output matches the selected parameters.

7. Validation & Visualization

  • Density Plots: Visualize distribution of tissue and sample_type to confirm lack of batch effects.

  • Jaccard Similarity: Plot a heatmap comparing ct_broad (ground truth/reference) vs. cell_type_estimation_idxmax_ora to validate cluster identities.

  • Marker Genes: Plot UMAPs for canonical markers (e.g., MS4A1, CD3D, ITGAX) to biologically verify cluster identities.

8. Saving

  • Save the fully clustered and annotated object to 03_tonsil_blood_DISCO_clustered.h5ad.

Clustering

Speed VS Determinism - Excourse on GPU Usage

Most GPU-accelerated functions are inherently non-deterministic due to their
asynchronous and parallelized execution. Operations are distributed across
many threads and cores, so the order of floating-point calculations can vary,
causing small numerical differences between runs. Libraries such as PyTorch,
TensorFlow, and cuML document this behavior. When using GPU-based algorithms
like UMAP, slight variations may occur each time. Although exact
reproducibility cannot be guaranteed, the computed embedding can be saved
and reused later to ensure consistent visualization. For Leiden clustering,
these variations are usually marginal and do not affect differential expression (DEG) analysis results.

PCA

PCA

PCA (Principal Component Analysis) reduces dimensionality by projecting data
onto directions (principal components) that capture the most variance. It is
extremely sensitive to outliers, which can strongly distort these
directions in a bad way. PCA also implicitly assumes data is normally distributed, which is rarely true in single-cell data.
Critically, downstream clustering results depend entirely on this projection,
so careful tuning and interpretation are essential.

PCA Parameter tuning

Dependent on sequencing depth, sample quality, number of cells (fewer reads per
cell if many cells are captured), cell-type heterogeneity, and many other
factors, there is no strict default for the number of principal components (PCs) to use.
A common and robust approach is to select the number of PCs that explain a
target cumulative variance — here, 95%, as stored in
adata.uns["config"]["general"]["pca_variance_sum"] = .95.
In this dataset, we have okish data quality, plus further deduplication
through demultiplexing, which justifies using the 95% variance threshold
confidently.

We had datasets where we could use 99.5% variance because the data quality was extremely high, and others where we had to limit it to 85% due to super noisy data, to don’t have the batch effect creating PCs.

This is a parameter you need to tune carefully depending on your dataset and analysis goals — do not blindly trust “best practices” or default recommendations.

Also the percentage is a guideline, you can also set it to a number of
PCs close to the one of the percentage. e.g 95% resulted in 14 pcs, you can try 13 by setting it manually
config["pp"]["neighbors"]["n_pcs"] = 14 > NOTE: if you rerun the sc_code.cluster_SC_scanpy_like(..., run_pca=True) it will again overwrite the config["pp"]["neighbors"]["n_pcs"]. But there is anyway no good reason to rerun pca unless you change the number of genes or cells

Neighborhood Graph, Leiden, UMAP

Neighborhood graph

The neighborhood graph is built on the PCA-transformed data and defines local
relationships between cells. It is highly sensitive to the number of PCs (``n_pcs``), since these determine how much variation is included in the
graph. Too many PCs can introduce noise, while too few can miss important
biological signals.
The graph also depends on the number of neighbors (``n_neighbors``), which
controls local connectivity. Smaller values emphasize fine, local structure;
larger values capture more global relationships. > NOTE:
> This is exactly the parameter you need to tune if you see strong batch effects. (reduce percentage)
> I have personally never had to apply batch effect correction and only observed
> a “batch effect” when the data was not supposed to be aligned in the first place.

Neighborhood graph Parameter tuning

The choice of n_pcs depends on sequencing depth, sample quality, cell number,
and biological heterogeneity. There is no strict default; it must be carefully
tuned to balance signal and noise via the explained variance of PCA.
The n_neighbors parameter defines the size of each cell’s local neighborhood.
Lower values highlight local details and can lead to many small communities;
higher values smooth out local variation and emphasize broader patterns.
Careful tuning of these parameters is essential, as the neighborhood graph
directly affects downstream clustering and embeddings (Leiden, UMAP).

UMAP

NOTE:
UMAP is only for visualization purposes and to help tune the n_neighbors
parameter for the neighborhood graph. Distances in the UMAP embedding do not
necessarily have any biological meaning, making it impossible to draw formal
conclusions based on point distances alone.
UMAP (Uniform Manifold Approximation and Projection) is a nonlinear dimensionality
reduction technique commonly used for visualizing high-dimensional single-cell data
in 2D or 3D space. It preserves both local and some global structure, making it
well-suited for exploring cell-type relationships and gradients.
UMAP depends heavily on the neighborhood graph and reflects its structure in the
embedding. Thus, any changes to PCA, n_pcs, or n_neighbors directly impact
UMAP results.

UMAP Parameter tuning

UMAP has two key parameters:

  • ``n_neighbors``:
    This is predetermined by the neighborhood graph and therefore shared with the leiden
    we can use the umap clustering to estimate a good n_neighbors parameter for the leiden,
    which is the thing we are actually interested in.
  • ``min_dist``:
    Determines how tightly points are allowed to pack together. Smaller values
    create more compact, separated clusters; larger values spread points more
    evenly. Actually, this is not the most critical parameter. We typically use
    around 0.5, unless collaboration partners specifically request a different
    UMAP appearance even though it doesn’t make any sense unless you need to
    “sell” your analysis as looking more impressive than it is.

Leiden Clustering

Leiden clustering operates entirely on the neighborhood graph and partitions
cells into discrete groups. It is a hierarcical-clustering graph-based community
detection algorithm that depends heavily on the underlying graph structure, the Neighborhood graph!

Leiden Clustering Parameter tuning

  • ``resolution``:
    The key parameter in Leiden is resolution, which controls clustering
    granularity. Higher resolution values produce more, smaller clusters; lower
    values merge cells into fewer, larger groups.
    Because clustering results depend fully on the neighborhood graph, tuning both
    the graph parameters and Leiden resolution is crucial for meaningful
    interpretation.
SUMMARY: Why tuning is important:
Each dataset has unique quality and biological and technical complexity. The Scanpy and
Seurat defaults are based on a small, pre-cleaned early 10x PBMC dataset
no clue why anyone would blindly trust those parameters. Proper tuning
captures true signals, minimizes noise, and ensures accurate clustering,
clear visualizations, and robust downstream analyses.

getting the cell_type_estimation_idxmax_ora

Here we use the sc_utils.add_cell_type_idxmax_from_significant.
It basically subsets the GSE estimates by p-values using
sc_utils.mask_gse_by_significance_threshold, and calculates the
nan-idxmax for cell types. Additionally, there are filters — check
the Doc for more information. It also saves the significant results
only in the obs if sig_out_key is True or a string (the obsm
key).
NOTE: Not all decoupler methods produce p-values, but we
encourage you to use the ones that have a certainty estimation.

import seaborn as sns
import matplotlib.pyplot as plt
import pandas as pd
import numpy as np
import scanpy as sc

def plot_ref_vs_idx_heatmap(
    adata: sc.AnnData,
    col_ref: str,
    col_idx: str,
    figsize: tuple[int, int] = (8, 6)
) -> None:
    """
    Plot heatmap of percentage distribution of `col_ref` across unique values
    in `col_idx`.

    Args:
        adata (sc.AnnData): Annotated data matrix.
        col_ref (str): Ground truth column in adata.obs.
        col_idx (str): Estimated label column in adata.obs.
        figsize (tuple[int, int], optional): Figure size. Defaults to (8, 6).

    Raises:
        ValueError: If columns are missing or contain invalid data.
    """
    if col_ref not in adata.obs or col_idx not in adata.obs:
        raise ValueError(f"Columns {col_ref} and/or {col_idx} not found in adata.obs")

    df = adata.obs[[col_ref, col_idx]].dropna()
    ctab = pd.crosstab(df[col_idx], df[col_ref], normalize='index') * 100
    ctab = ctab.round(2)

    plt.figure(figsize=figsize)
    sns.heatmap(ctab, annot=True, fmt=".1f", cmap="viridis", cbar_kws={'label': '%'})
    plt.xlabel(col_ref)
    plt.ylabel(col_idx)
    plt.title(f"Distribution of {col_ref} (%) across {col_idx}")
    plt.tight_layout()
    plt.show()

plot_ref_vs_idx_heatmap(
    adata,
    col_ref = "ct_broad",
    col_idx = "cell_type_estimation_idxmax_ora")
